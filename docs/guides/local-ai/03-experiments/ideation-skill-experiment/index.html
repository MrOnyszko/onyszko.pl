Ideation: Audio Notes with Transcription

Date: 2026-03-27 Status: Draft | Ready for PRD

Problem#

What problem are we solving?#

Users lose ideas while walking because they can’t type, and currently have no way to record audio notes that automatically transcribe to text within Aily.

Who has this problem?#

People who capture ideas while walking/moving, and users who need to transcribe voice notes (meetings, conversations).

How do they solve it today?#

Using Google Recorder app to record audio and get transcripts, then manually copying the text back into Aily. This is fragmented and loses context.

Root Cause Hypothesis#

Aily currently only supports text notes, forcing users to leave the app to capture voice ideas, breaking their flow and causing lost ideas.

Solution#

Proposed approach#

Add audio recording capability to notes with automatic on-device transcription. Users can record while walking, get a transcript, edit it, and play back the audio alongside the text.

MVP behavior#

  • Record audio directly from the note creation/edit screen
  • Transcribe audio to text using local on-device transcription (Cactus)
  • Display transcript with ability to edit
  • Play back audio alongside the transcript
  • Store both audio file and transcript text in the note

What’s NOT in scope#

  • Cloud-based transcription
  • Multi-language support (start with single language)
  • Audio editing or trimming
  • Sharing audio files
  • Real-time transcription display during recording

Value#

Why is this worth building?#

Captures ideas in the moment when users can’t type, reduces friction in note-taking, and keeps users within Aily instead of switching apps.

Success criteria#

  • Users can record, transcribe, and save a note in under 30 seconds
  • Transcription accuracy is >90% for clear speech
  • Users create notes using audio at least 20% of the time in the first month

Early signal metric#

Percentage of new notes created with audio attachment

Risks#

Risk Likelihood Impact How to Validate
Transcription quality is poor Medium High Test with real users in various environments
Transcription is slow High High Benchmark Cactus performance on target devices
Storage bloat from audio files High Medium Estimate storage per note, set size limits
Recording UX is clunky Medium Medium User testing of recording flow
On-device transcription not fast enough High High Prototype and measure latency

Open Questions#

  • Which languages to support at launch?
  • What’s the maximum audio file size to allow per note?
  • How to handle transcription errors in the UI?
  • Should users be able to re-transcribe after editing?
  • What’s the default storage location for audio files (local vs app storage)?
  • How to display audio waveform in the note viewer?
  • Should transcription happen immediately or on-demand?
  • What permissions are needed (microphone, storage)?
  • How to handle transcription failures gracefully?
  • Should there be a “record while typing” mode?

Next Steps#

  • Prototype Cactus integration and measure transcription speed/quality
  • Design recording UX flow (start/stop, visual feedback, error states)
  • Define audio file storage strategy and size limits
  • Write the PRD using the create-prd skill
