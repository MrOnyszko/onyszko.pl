09 - PRD Creation Experiment

  • Open in ChatGPT
  • Open in Claude
  • Open in Perplexity
  • Copy Markdown

In this post, I will show you how to use local AI models to generate Product Requirement Documents that developers can act on directly.

Why#

Writing good requirements is hard. Writing good requirements consistently is even harder. Most developers skip PRD creation entirely and jump straight to coding — then wonder why scope creeps, priorities blur, and features don’t quite fit together. Using AI for PRD creation gives you three things that manual processes don’t: discipline (the model forces you to answer specific questions), speed (minutes instead of hours), and consistency (each PRD follows the same structure). The local LLM aspect means you get this without sending your product ideas to the cloud.

How#

What is a PRD?#

A Product Requirement Document is a structured artifact that describes what you’re building, why you’re building it, and how you’ll know it’s done. It’s not a spec — it’s the contract between product thinking and engineering execution.

A good PRD answers:

  1. What — The feature or product and its core value proposition
  2. Who — The target user and their problem
  3. Why — The business case and success criteria
  4. How — The functional requirements and user flows
  5. When — Timeline and milestones

A PRD is not a technical spec. Don’t include database schema or API endpoints — save that for later.

Why PRD is Useful#

The discipline of writing a PRD catches problems before you write code:

  • Scope clarity — “It should do X” becomes “It must do X, Y, Z and NOT A, B.” That’s a 15-minute conversation that saves 15 days of rework.
  • Dependency mapping — When you write down what the feature needs, you see what other systems it touches.
  • Testable acceptance criteria — A PRD that says “users can create tasks” is useless. A PRD that says “users can create tasks with title, due date, and priority; tasks appear in list sorted by due date” gives QA something concrete.
  • Onboarding aid — New developer joins the team? Drop the PRD in Slack and they’re up to speed in 10 minutes.

Let’s Use Gemini Pro to Rate Our PRDs#

I fed both PRDs into Gemini Pro and asked it to rate them using a standard PRD evaluation rubric.

Gemini’s rating criteria:

  1. Clarity (1-5): Is the feature and its value proposition clear?
  2. Completeness (1-5): Are all required sections present?
  3. Actionability (1-5): Could a developer implement from this without additional clarification?
  4. Testability (1-5): Are acceptance criteria specific enough to verify?
  5. Consistency (1-5): Do requirements align with user stories?

Let’s Create an Agent for PRD Creation#

Based on the experiments above, I created an OpenCode skill that automates PRD creation. It uses the question tool to gather requirements systematically, then generates a structured PRD.

PRD Agent skill location: ~/.config/opencode/skills/prd-creator/SKILL.md

The skill uses OpenCode’s question tool to iteratively gather requirements — the same pattern you’d use in a real product conversation.

Key Points Recap#

  1. PRDs force discipline — The structured format catches scope ambiguity before code gets written
  2. Local LLMs generate usable PRDs — Qwen3.5-35B-A3B produces 80% of the quality at 0% cost
  3. Cloud models may add polish — MiniMax M2.5 showed slight improvements in completeness
  4. External rating reveals blind spots — Gemini’s evaluation caught gaps I missed in my review
  5. Agents make it repeatable — The PRD Creator skill turns one-off generation into a reusable workflow
